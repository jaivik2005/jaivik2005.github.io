Git & GitHub
Complete Guide

From Beginner to Advanced. The reality of how 95%+ of companies use Version Control.


What is Git?

Git is a **Distributed Version Control System (DVCS)**.

🔍 Simple Meaning:

Imagine you are writing a massive book. Every time you finish a chapter, you want to save a "snapshot" so you can go back if you mess up later. Git does exactly that for code.

Git helps you:

  • Track every single code change (who changed what, and when).
  • Go back to previous versions instantly if something breaks.
  • Work with teams without conflicts (multiple people editing same files).
  • Keep your project safe & organized.

💡 Think of Git like: Google Docs for code, but offline + super powerful.

📌 What is GitHub?

GitHub is a cloud platform that hosts Git repositories. It's like the social network for developers.

🔍 Git vs GitHub (Very Important)

Many beginners confuse these two. Here is the clear difference:

Git GitHub
Tool (software) installed on PC Website / Cloud Platform
Works offline (Local) Needs internet (Remote)
Version control engine Hosting + Collaboration Tools
Used via terminal (Command Line) Used via browser

Industry Usage (REALITY)

In the professional world, Git is not optional. It is the standard.

✅ Where Git is USED (95%+ companies)

  • Software Companies: Google, Microsoft, Amazon, startups.
  • Government IT: NIC, ISRO, DRDO use it for secure code management.
  • Freelancing: Clients expect you to push code to their GitHub.

  • Very small one-person non-technical work.
  • Jobs that only use MS Excel / Word.
  • Legacy systems (Mainframes) - though even they are migrating.
👉 Reality: If you don’t know Git → You generally cannot get a modern developer job.

How Git Works (Internals)

Git doesn't just "save" files. It moves them through 3 main areas:

  1. Working Directory (Writing Exam): This is your folder where you are currently editing files. It's like doing rough work.
  2. Staging Area (Final Answers): These are files you have marked as "ready to save". You select specific files to move here using `git add`.
  3. Repository (Submitted Sheet): This is the permanent history. When you `git commit`, files move from Staging to here.

Installation & Config

Installing Git

Windows: Download "Git Bash" from git-scm.com.

Linux (Ubuntu/Debian):

sudo apt install git

Check if it works:

git --version
# Output: git version 2.43.0

⚙️ First-Time Configuration (MANDATORY)

Git needs to know who is making changes. This "tags" every commit with your identity.

git config --global user.name "Your Name"
git config --global user.email "youremail@gmail.com"

Check your settings:

git config --list

Basic Commands Explained

These 6 commands cover ~80-90% of real industry usage. Here's exactly what they do.

1. git init (Start)

Creates a new Git repository. It creates a hidden .git folder that tracks everything.

git init
# Output: Initialized empty Git repository in /project/.git/

2. git clone (Copy)

What it does: Downloads a copy of a remote GitHub repository to your computer.

Used when:
  • Joining a new team or project.
  • Downloading open-source code.
git clone https://github.com/user/project.git

3. git status (Check Status)

MOST USED COMMAND. Checks what has changed. If you are confused, run this.

git status
# Output:
# On branch main
# Untracked files: index.html

4. git add (Staging)

Moves files to the "Staging Area" (ready to be saved).

git add index.html   # Add one file
git add .            # Add ALL changes (Standard)

5. git commit (Save)

Saves a snapshot of your project history. Always write a message!

git commit -m "Added login page layout"
# Output: [main 95bfa60] Added login page layout

6. git log (History)

Shows a list of all save points (commits).

git log

Branching Strategy (Must Know)

Why branches? Imagine working on a "Dark Mode" feature. You don't want to break the main website while testing. You create a separate "branch", write code there, and merge it only when it's perfect.

Classic Commands

git branch feature-x Create a new branch
git checkout feature-x Switch to that branch
git checkout -b feature-x Create & Switch in one step (Pro tip!)

Merging (The Reunion)

When you are done, you bring your changes back to the main code.

git checkout main        # Go back to main
git merge feature-x      # Pull changes from feature

⚠️ Merge Conflict (Real Life)

Occurs when two people edit the exact same line. Git panics and asks you to choose.

CONFLICT (content): Merge conflict in index.html

Fix: Open the file, choose the correct code, delete the `<<<<` markers, then `git add .` and `commit`.

GitHub Workflow (Industry Standard)

Step-by-Step Cycle:

  1. Clone repo to your laptop.
  2. Create a new branch (e.g., `feature-payment`).
  3. Write code, Add, and Commit.
  4. Push branch to GitHub.
  5. Open a Pull Request (PR) on GitHub website.
  6. Team reviews code (comments/approval).
  7. Merge PR into main.

Key Remote Commands

1. git push (Upload)

Sends your local commits to GitHub.

git push -u origin main

2. git pull (Update - Very Important)

Downloads new changes your team made. Run this every morning!

git pull origin main

Git + Deep Learning / AI

Why Git is CRITICAL in AI?

In AI/ML, you don't just version code. You version experiments.

  • Dataset Versioning: Knowing exactly which data trained a model.
  • Model Experiments: "Try ResNet50 vs EfficientNet" -> distinct branches.
  • Reproducibility: Ensuring a paper's result can be recreated 6 months later.

Typical AI Repo Structure

project/
 ├── data/            # (Ignored) Raw datasets - never push GBs of data!
 ├── notebooks/       # (Experiments) Jupyter notebooks for testing
 ├── models/          # (Ignored or tracked via LFS) Trained .pth/.h5 files
 ├── src/             # (Clean Code) Python scripts for training/inference
 ├── requirements.txt # Dependencies
 └── README.md        # Instructions

Advanced Git (Senior Level)

These commands are not used daily, but they save you from heavy disasters.

1. git rebase (Clean History)

What it does: Rewrites history to make it a straight line, removing messy merge commits.

Analogy: "Replay my work as if I started it today."

git rebase main
⚠️ WARNING: Never rebase a public branch that others are working on!

2. git cherry-pick 🍒 (The Sniper)

What it does: Picks ONE specific commit from another branch and copies it to yours.

Scenario: A critical bug was fixed in `dev` branch. You need THAT fix in `prod` immediately, but not the rest of the unfinished code.

git cherry-pick a1b2c3d

3. git stash pop (The Pocket)

What it does: Temporarily saves your unfinished work so you can switch branches.

git stash        # Put work in pocket
git checkout other-branch
# ... do stuff ...
git checkout original-branch
git stash pop    # Take work out of pocket

4. git bisect 🔍 (The Detective)

What it does: Uses binary search to find exactly which commit broke the code. It's magic for old bugs.

git bisect start
git bisect bad   # Current version is broken
git bisect good <old-commit-hash> # It worked back then

Security & Rules

⚠️ What NOT to Push (Industry Rule)

Use a .gitignore file to strictly exclude:

node_modules/    # Heavy dependencies
.env             # API Keys & Secrets (CRITICAL)
__pycache__/     # Python cache
data/            # Large datasets
models/          # Large binary model files

🔐 Best Practices

  • Never push passwords. Use environment variables.
  • SSH Keys: Use SSH instead of HTTPS for secure, password-less pushing.
  • Small Commits: "Fix login button" is better than "Updated entire website".

Interview Questions (REAL)

Q: Git vs GitHub?
Git is the tool (version control), GitHub is the hosting platform.

Q: What is a commit?
A snapshot of your project. It saves the state of all files at that moment.

Q: Why branch?
To allow parallel development so multiple people can work without breaking the main stable code.

Q: What is a PR (Pull Request)?
A request to merge your branch into the main codebase, usually requiring code review.
